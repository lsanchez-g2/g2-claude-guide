        Before you read on

        Two words you'll see everywhere below.

        Both stretch across all four products. Worth a quick read before you dive in.

        Connectors

        The bridge to your other tools.

        Gmail, Notion, Slack, Asana, GitHub, Google Drive, Dovetail. Connect them once in Settings, and any Claude product can read and write through them. You'll see them referenced as MCP (Model Context Protocol) in some docs. Same thing.

        Examples in this guide: Chat using the Asana connector to build a live task dashboard. Cowork using Gmail to sweep receipts. Code using Jira to pick the next ticket.

        Plugins

        One-click bundles you install on top of Claude.

        A plugin can pack skills, connectors, slash-commands, and sub-agents together for a specific job. Anthropic publishes plugins for sales, legal, finance, marketing, support, product, and design. Your team can also publish private plugins for internal workflows that don't belong on the open marketplace.

        If a skill is one instruction, a plugin is a kit. Think "designer starter pack" instead of "one design tip."

In 2026 it stopped being just a chat box: Artifacts evolved from one-off HTML previews into persistent, connector-backed workspaces, Skills give Claude reusable instructions for your workflows, and Projects keep long-running work organized.

        When to reach for Chat, and when not

        Use Chat when

        • The conversation is the work. Thinking, drafting, summarizing, asking.
        • You want a re-openable view. Live Artifacts beat static reports for anything that changes.
        • Your data lives behind connectors. MCP-backed Artifacts read and write live.

        Reach elsewhere when

        • The work touches local files. Use Cowork. Chat can't reorganize your Downloads folder.
        • You need a real export. Use Design for PPTX / Canva / PDF with proper typography.
        • You're running multi-agent code work. Use Code. it owns the terminal, the diff, and the PR.
        Author's pick · don't conflate Artifacts with Design

        Artifacts are conversation-scoped live HTML; Claude Design is a persistent visual canvas. Use Artifacts when you'd reach for a code sandbox: quick dashboards, mini-tools, working prototypes. Use Design when you'd reach for Figma or Canva: real exports, multiplayer review, a build-handoff bundle. They overlap on "show me something visual", but the export paths differ, and that's usually the deciding factor.

        Deep dives

        PMChoosing between an Artifact and a Project

        An Artifact is a single rendered surface: one page, one chart, one tool. A Project is the workspace around the artifacts: file context, instructions, memory. If the work is "one view that updates," it's an Artifact. If it's "a long-running effort with multiple artifacts and reference files," it's a Project that contains Artifacts.

        Practical decision: start in a Project the moment a second conversation about the same thing happens. The cost is zero, the benefit is recovered context.

        DesignerWhat an Artifact can render, and where it breaks down

        Artifacts ship raw HTML, React, SVG, and Mermaid. They're great for working prototypes like a date picker, a mini dashboard, or a pricing calculator. They're not a typography surface, though. Web fonts work, but you don't have a layout panel, you can't drop frames on a canvas, and you can't export to PPTX or Canva. For anything that needs to leave the browser as a polished deliverable, hop to Design.

        The honest mental model: Artifacts are a code sandbox glued to the conversation; Design is the surface that owns "I want to make a thing that looks made."

        EngineerMCP, Artifact storage, and the Live data model

        Artifacts get up to ~20 MB of persistent storage per artifact and can call MCP tools directly through the connector layer (Pro+/Team/Enterprise). On reopen, a Live Artifact transparently caches reads and exposes window.cowork.callMcpTool(name, args)-style hooks (the exact API surface differs between Chat Artifacts and Cowork Artifacts; check the docs site map before building).

        The pattern for stability: treat each fetch as idempotent, cache by query, never store secrets in the artifact body, and put any user-mutating action behind an explicit click.

        04 ·Claude Cowork3 min read

        A teammate that opens your folders.

        Your desktop teammate. Reads, edits, and creates files in folders you authorize. Runs multi-step work in the background.

        Hand Cowork a messy folder and a goal. It opens, reads, classifies, renames, and files with explicit per-folder permission.

        Outcome Multi-step file work done on your computer, not yours to do
        Tier Pro · Max · Team · Enterprise macOS & Windows
        Time-to-value ~5 minutes from install to first reorg

        The Claude that lives where your files do.

        Cowork launched in research preview on January 12, 2026, and reached general availability on April 9. It's cross-platform on macOS and Windows for every paid plan. The promise is small but consequential: Claude can finally do the boring desktop work. Receipt sorting, note digesting, screenshot OCR, weekly digest generation, all without you driving every step.

        The mental model is "teammate with read/write access to one folder." Access is granular, explicit, and revocable. Cowork combines Skills and connectors with local filesystem operations, so it can blend your Asana tasks, your Gmail, and the PDFs in ~/Downloads in a single run.

        Capability Folder-level permissions

        You grant Cowork access per folder, not per session. Revoke any time. The permission UI is the same in macOS and Windows.

        Capability Multi-step file work

        Organize, rename, split, merge, OCR, extract, summarize. Tasks chain across files; long-running work surfaces progress.

        Capability Scheduled tasks

        "Each morning, do X." Cowork's scheduler triggers tasks on cron-like cadences or one-off times. Combine with local files + MCP.

        Capability Skills & plugins

        Same skill format as Chat. Install Anthropic-published or community plugins (sales, legal, marketing, finance, support) to extend behavior.

        Capability Connector + local blend

        A run can read Gmail, hit Asana, and write to ~/Documents, connectors and the filesystem in one workflow.

        Capability Persistent Artifacts

        Cowork Artifacts persist across sessions, can call MCP tools, and remember user choices in localStorage, useful for personal trackers.

        When to reach for Cowork, and when not

        Use Cowork when

        • The work touches local files. Folders, batches, OCR, renames, splits, merges.
        • It needs to run on a schedule. Daily digests, morning sweeps, end-of-week summaries.
        • You want autonomy. Multi-step jobs with progress, not turn-by-turn chat.

        Reach elsewhere when

        • The data lives only in connectors. Use Chat with MCP Artifacts, no local files involved.
        • You're working on a real codebase. Use Code. diff hygiene, PRs, test runners, subagents.
        • You need a polished visual export. Use Design for Canva/PPTX/PDF with intent.
        Author's pick · grant the smallest folder that works

        Don't give Cowork your home directory. The permission UX is per-folder for a reason. Start with the narrowest folder that contains the work. ~/Downloads/receipts/ beats ~/Downloads; ~/Documents/clients/acme/ beats ~/Documents. You'll get fewer surprising actions, faster reasoning over the file set, and a safer audit trail. Expand access only when the task forces you to.

        Deep dives

        PMCowork vs. Chat.same Skills, different surface

        Both surfaces share Skills and connectors. The difference is filesystem access. If your team's workflows live in Google Drive, Notion, Asana. Chat with MCP is enough. If they live in Dropbox-synced folders, Finder, or "the share drive," Cowork wins because it can act on the bytes, not just read them.

        Decision shortcut: if the work mentions a filename or a path, use Cowork.

        DesignerPersistent Artifacts that remember filter state

        Cowork's Artifacts persist across app restarts and can use localStorage to remember the user's filter and sort choices. Useful for personal trackers (interview pipeline, content calendar, design-review queue). Combine with window.cowork.askClaude(prompt, data[]) for inline Haiku inference over the data you just fetched.

        The header gets a Reload button for free, so don't build your own.

        EngineerBuilt largely with Claude Code, in roughly 10 days

        Anthropic engineers built Cowork in about a week and a half, mostly by driving Claude Code itself. The motivating signal was watching non-engineers use Claude Code for non-programming tasks like taxes, receipts, and organizing notes. The conclusion: the right abstraction is a folder, not a repo.

        For integrators: Cowork exposes callMcpTool, askClaude, and runScheduledTask on window.cowork inside Artifacts. Treat connectors as read-mostly; gate writes behind explicit user clicks.

        When to reach for Code, and when not

        Author's pick · write a CLAUDE.md on day one

        The single highest-leverage move is writing a CLAUDE.md before your first real task. Even a 20-line file.preferred libraries, the test command, the review checklist, the "don't touch" modules.multiplies output. Treat it like onboarding documentation for a new hire who reads it every morning. The 2-4× velocity numbers people report come almost entirely from teams that invested in this config, not from raw model capability.

        Deep dives

        PMHow to estimate Claude Code's effect on a team

        Published reports of 2-4× velocity come from teams that invest in two things: a maintained CLAUDE.md and a subagent workflow for high-fan-out tasks. Teams that drop Claude Code into a repo with no config see far smaller gains.

        For a baseline: track PR throughput, time-to-first-review, and percentage of PRs requiring fewer than 3 review rounds. Re-measure 4 weeks after rollout. Adoption is the variable, not the model.

        DesignerWhat Code produces that designers consume

        Three artifacts worth asking Code to produce explicitly: a component inventory (everything the codebase renders), a token export (CSS variables, design tokens, color values), and a component story file (Storybook stub or equivalent). These pair cleanly with Claude Design's handoff bundle on the way back in.

        Designer-friendly prompt: claude "Produce a JSON inventory of every component, its props, and the tokens it consumes."

        EngineerSubagents, hooks, MCP.the pattern that scales

        Subagents work best when the task fans out (per-module refactors, per-test fixes, per-file documentation). The lead agent owns planning & merging; workers own execution. Don't fan out for sequential work.the merge cost dominates.

        Hooks are where you encode safety: a pre-tool hook that blocks writes to infra/; a post-tool hook that runs linters after every file write; an on-error hook that posts to Slack via MCP.

        MCP servers are the right place for any tool you'll reuse.pull tickets, push releases, fetch logs. Write the schema thin, gate side-effects behind explicit approvals, version the server like any other dependency.

        Deep dives

        PMBuying decisions. Design vs Canva vs Figma

        Design isn't a Figma replacement. It's a way to generate, iterate, and export visual deliverables without a designer in the loop for the first pass. Teams that have Figma keep Figma; Design accelerates the first 60%.the part that usually never gets done because no one had the bandwidth to start.

        The Canva partnership is a hint about the intended buyer: people who already think in slides and one-pagers, not in components.

        DesignerHow the team design system gets built

        On first onboarding, point Design at your codebase + your existing design files. Claude reads both, infers tokens (colors, type, spacing), and assembles a team design system you can edit before locking. Subsequent projects reuse this system automatically; tokens propagate, components stay in family.

        Practical tip: clean the design files before onboarding. Junk in, junk system out. The onboarding step is your one chance to shape the inferred system without manual override.

        EngineerThe handoff bundle, file by file

        The Code handoff bundle is four artifacts:

        spec.json. component structure as a machine-readable spec (component name, variant, props, slot tree).

        tokens.css. the design tokens used on the canvas, exported as CSS custom properties. Drop directly into a stylesheet or import via your token pipeline.

        layout.tsx. layout hierarchy expressed as React (other frameworks coming). Replace with your own primitives; the bundle gives you the tree shape.

        /assets/. referenced images, icons, fonts. Hash-named, deduped.

        Read the bundle in claude with: claude "wire ./bundle into apps/web/src/routes/dashboard". Claude Code uses the spec to scaffold the route, the tokens to extend the theme, the layout to seed JSX, and the assets to wire imports.

        07 ·Playbooks

        Three workflows that stitch the stack.

        Three named routes that move work between products. Each one stitches together a flow you'd otherwise hand-roll.

        01 Spec → Ship

        A PM's one-pager becomes a shipped PR.

        Draft, visualize, hand off, ship. No copy-paste between tools or hand-rolled translation between disciplines.

        1. 01 Chat Draft a one-pager PM writes the goal, success metric, and constraints in a Project.
        2. 02 Design Visualize on canvas Designer turns the one-pager into a slide deck, then a clickable prototype.
        3. 03 Design Export handoff bundle spec.json · tokens.css · layout.tsx · /assets/.machine-readable.
        4. 04 Code Wire & ship the PR Engineer runs claude with the bundle. Claude scaffolds the route, opens the PR.

        02 Inbox → Insight

        A messy quarter becomes a re-openable dashboard.

        Receipts, notes, and screenshots get tidied on disk; a connector-backed Live Artifact summarises the result and stays current.

        1. 01 Cowork Sort the folder A run organizes ~/Downloads into receipts / invoices / screenshots.
        2. 02 Cowork Generate a digest A scheduled task writes a CSV summary plus a Markdown brief next to the folder.
        3. 03 Chat Build a Live Artifact Chat plots the CSV through the connector and exposes it as a re-openable page.
        4. 04 Chat Share with leadership Pinned to a Project. Re-fetches on every open.no manual refresh.

        03 Repo → Refactor

        An unfamiliar codebase becomes a stakeholder slide.

        Engineering maps the repo with subagents; a one-pager flows out of the findings; a slide deck closes the loop with leadership.

        1. 01 Code Fan-out repo map Subagents partition the repo; lead agent merges into a single risk-tier report.
        2. 02 Chat Write the one-pager Findings get pasted into a Project; Chat distils them into an Artifact memo.
        3. 03 Design Promote to a slide Design lifts the memo onto a deck using the team design system.
        4. 04 Design Export PPTX Leadership review deck, formatted, brand-correct, editable on hand-off.
        Author's pick · the boring shortcut

        Don't try to use all four at once on your first run. Pick a single playbook, run it twice end-to-end, and only then add the next product. The friction in cross-product work isn't capability.it's habit, and habits take a real reps to land. The Spec → Ship playbook is the cleanest first one for most teams; Inbox → Insight is the most underrated.
